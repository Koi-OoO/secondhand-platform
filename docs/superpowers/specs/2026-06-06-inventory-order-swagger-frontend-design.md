# 库存下单与接口联调设计

## 目标

为当前二手交易平台增加“可售库存”和“按数量下单”能力，并以此为基础完成：

1. 后端库存与订单逻辑改造。
2. 在线 Swagger / OpenAPI 文档同步更新。
3. 基于文档的接口联调测试。
4. 前端发布页、详情页、订单页与后端接口语义对齐。

本次以“最小可闭环方案”为准，不引入支付系统、库存锁定超时、SKU、多规格等扩展能力。

## 当前现状

### 数据模型

- `product` 表没有库存字段。
- `order` 表没有购买数量字段。
- 当前商品状态依赖 `status`：
  - `1` 在售
  - `2` 已下架
  - `3` 已售出

### 下单逻辑

- `POST /order/create` 只接收 `productId` 和 `address`。
- 下单成功后直接把商品状态改为 `3`，默认每个商品只有 1 件。
- 买家取消订单、卖家拒绝发货时，会把商品状态恢复为 `1`。

### 前端现状

- 商品发布页没有库存输入。
- 商品详情页没有数量选择器。
- 订单页没有数量展示。
- Swagger 文档尚未覆盖库存和数量相关语义。

## 业务规则

本次按以下规则实现：

1. 商品支持库存数量。
2. 卖家发布商品时必须填写库存。
3. 买家下单时必须选择购买数量。
4. 下单成功立刻扣减库存。
5. 取消订单或卖家拒绝发货时恢复库存。
6. 库存减到 `0` 时商品变为不可购买。
7. 当前仍不引入真实支付流程，因此订单创建后直接进入现有的待发货链路。

## 方案选择

### 方案 A：商品主表直接存库存

- 在 `product` 表增加 `stock` 字段。
- 在 `order` 表增加 `quantity` 字段。
- 继续复用现有订单状态和商品状态。

优点：

- 改动集中。
- 容易与现有控制器、服务、前端对接。
- 最快形成完整可测试闭环。

缺点：

- 不适合未来多规格或复杂库存流水。

### 方案 B：独立库存表

- 商品主表不直接存库存。
- 引入库存表或库存流水表。

优点：

- 扩展性更强。

缺点：

- 超出当前项目复杂度。
- 会显著放大 Swagger、前端和测试改动面。

### 结论

采用方案 A。

## 数据设计

### `product` 表

新增字段：

- `stock INT NOT NULL DEFAULT 1`

含义：

- 表示当前可售库存。
- 发布商品时由卖家填写。
- 编辑商品时可修改。

校验规则：

- `stock >= 1`

### `order` 表

新增字段：

- `quantity INT NOT NULL DEFAULT 1`

含义：

- 表示当前订单购买的商品数量。

校验规则：

- `quantity >= 1`

## 商品状态定义

继续保留现有 `product.status`，但语义明确为：

- `1` 在售：卖家上架且库存大于 0
- `2` 已下架：卖家主动下架，不可购买
- `3` 已售罄：库存为 0，不可购买

说明：

- 本次将原来的“已售出”统一解释为“售罄/库存为 0”。
- 这是为了在不重构状态体系的前提下兼容库存模型。

## 后端接口设计

### 商品发布

接口：

- `POST /product/publish`

新增请求字段：

- `stock`

规则：

- 必填
- 必须为正整数

成功后：

- 商品初始状态为 `1`
- 商品库存为卖家填写值

### 商品编辑

接口：

- `PUT /product/update`

新增请求字段：

- `stock`

规则：

- 若传入则必须为正整数
- 编辑时仍保留当前约束：先下架，再编辑

状态联动：

- 若编辑后 `stock > 0`，状态保持由上/下架接口控制
- 本次不允许把库存改成 `0`
- 卖家若不想出售，应使用下架而不是把库存改空

### 商品详情

接口：

- `GET /product/detail/{id}`

新增返回字段：

- `stock`

说明：

- 前端据此展示剩余库存和购买上限。

### 创建订单

接口：

- `POST /order/create`

参数从：

- `productId`
- `address`

改为：

- `productId`
- `quantity`
- `address`

校验：

1. 用户已登录。
2. 商品存在。
3. 商品状态为 `1`。
4. 购买数量为正整数。
5. 购买数量不能超过当前库存。
6. 不能购买自己发布的商品。

订单生成规则：

- `order.quantity = quantity`
- `order.productAmount = 商品单价 * quantity`
- `order.freight = 0`
- `order.totalAmount = 商品单价 * quantity`

库存扣减规则：

- `product.stock -= quantity`
- 若扣减后库存为 `0`，则 `status = 3`
- 若扣减后库存仍大于 `0`，则状态保持 `1`

### 取消订单

接口：

- `PUT /order/{id}/cancel`

库存恢复规则：

- 订单取消成功后，`product.stock += order.quantity`
- 若商品原状态为 `3` 且恢复后库存大于 `0`，则状态恢复为 `1`

### 卖家拒绝发货

接口：

- `PUT /order/{id}/reject`

库存恢复规则：

- 与取消订单一致：
  - `product.stock += order.quantity`
  - 若恢复后库存大于 `0`，则状态恢复为 `1`

## 服务层改造要求

### 商品服务

需要修改：

- 商品发布校验
- 商品编辑校验
- 商品详情返回结构
- 商品缓存失效后的库存字段一致性

关键要求：

- 发布和编辑都要校验 `stock`
- 商品详情缓存中必须包含最新库存
- 发布、编辑、上下架、下单、取消、拒绝后都要清理商品详情缓存

### 订单服务

需要修改：

- 下单逻辑
- 取消逻辑
- 拒绝发货逻辑
- 订单列表展示数据

关键要求：

- 订单必须保存购买数量
- 订单列表和详情输出需包含 `quantity`
- 下单、取消、拒绝发货时的库存变更必须和订单状态变更放在同一事务中

## 并发与一致性

本次采用“与现有代码风格一致的保守方案”：

1. 下单时对商品库存扣减增加并发保护。
2. 优先复用现有 Redis / Redisson 能力。
3. 库存扣减与订单创建必须在同一事务中完成。
4. 不能出现库存被扣但订单未创建，或订单创建成功但库存未扣的情况。

建议实现方向：

- 对下单操作引入商品级分布式锁，和当前上下架逻辑保持一致。

## Swagger / OpenAPI 设计

需要同步更新以下文档内容：

1. `Product` 模型增加 `stock` 字段说明。
2. `Order` 模型增加 `quantity` 字段说明。
3. `POST /product/publish` 请求示例增加 `stock`。
4. `PUT /product/update` 请求示例增加 `stock`。
5. `POST /order/create` 参数增加 `quantity`。
6. 订单响应示例体现数量与金额语义。

验收标准：

- `/v3/api-docs` 输出中可以看到新字段。
- Swagger UI / Knife4j 页面中可以直接看到并测试新接口参数。

## 接口测试设计

后端完成后，至少覆盖以下用例：

1. 发布商品时传入库存，返回成功。
2. 商品详情能看到库存字段。
3. 用库存为 `5` 的商品下单 `2` 件，返回成功。
4. 下单后商品库存变为 `3`。
5. 再下单超出库存的数量，返回失败。
6. 取消订单后库存恢复。
7. 卖家拒绝发货后库存恢复。
8. Swagger 文档中可看到 `stock` 和 `quantity`。

## 前端页面改造

### 发布页 / 编辑页

页面：

- `ProductPublishPage.vue`

改造点：

- 增加库存输入框
- 表单校验：
  - 必填
  - 正整数
- 发布和编辑请求体带上 `stock`

### 商品详情页

页面：

- `ProductDetailPage.vue`

改造点：

- 展示库存信息
- 增加购买数量选择器
- 购买按钮提交时带上 `quantity`
- 当前端库存为 `0` 或状态不是 `1` 时，禁用购买

### 我的订单页

页面：

- `MyOrdersPage.vue`

改造点：

- 展示购买数量
- 展示金额时明确数量和小计关系
- 状态提示文案与库存模型保持一致

## 非目标

本次明确不做：

1. 支付系统
2. 下单占库存超时释放
3. 购物车
4. SKU / 多规格
5. 后台库存流水审计
6. 卖家批量改库存工具

## 验收标准

满足以下条件才算完成：

1. 后端编译通过，相关测试通过。
2. 商品支持库存发布与编辑。
3. 买家可按数量下单。
4. 下单后库存正确扣减。
5. 取消订单和拒绝发货后库存正确恢复。
6. Swagger 在线文档已同步反映字段和接口变更。
7. 基于文档的接口测试通过。
8. 前端发布页、商品详情页、订单页已对齐新接口语义并联调通过。
